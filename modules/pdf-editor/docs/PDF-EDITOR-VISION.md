## The Business Problem

Every small business that handles documents — purchase orders, inspection reports, compliance forms, permits, contracts — hits the same wall: **PDF tools are either expensive, clunky, or both.**

Adobe Acrobat costs $240/year per seat. Free tools lose your work, lack form support, and require uploading sensitive documents to third-party servers. Field workers on tablets struggle with tools designed for desktop. Form fields misalign. Drafts vanish on refresh. Multi-person signing chains are slow and confusing.

The result: people print PDFs, mark them up by hand, scan them back in, and email them around. In 2026.

The businesses that need this most — contractors approving POs, inspectors filling compliance forms, managers signing off on work orders — are small teams that can't afford enterprise document management but desperately need to mark up, fill, and generate PDFs reliably.

## What the Module Does

The Fireproof PDF Editor is a **stateless PDF processing engine for annotation rendering, form templating, form filling, and PDF generation.** It does not store PDF files. The caller — whether a human using the web UI or an app calling the API — provides PDF bytes, the editor processes them, and returns the result.

It answers three questions:
1. **What marks need to go on it?** — The caller sends PDF bytes + annotation data (stamps, text, shapes, arrows, signatures, highlights). The editor burns them into the PDF and returns the processed bytes.
2. **What data needs to be collected?** — Form templates define reusable field layouts (positions, types, validation rules). Submissions fill those fields with validated data. Templates and submissions persist in Postgres.
3. **What's the final output?** — The caller sends PDF bytes + a submission ID. The editor fills the template fields into the PDF and returns the generated bytes.

**The editor never stores PDF files.** In standalone mode, the user opens a PDF from their computer, edits it in the browser, and saves it back to their computer. In API mode, the calling app provides PDF bytes and decides where to store the result.

**Two modes of operation:**
- **Standalone:** Own web UI (React + Vite) for direct use — no other app required. Open from computer, annotate/fill in browser, save back to computer.
- **API integration:** Any vertical app sends PDF bytes to the module's REST API for processing (render annotations, fill forms, generate PDFs). The app manages its own file storage.

## Who Uses This

### Document Owner / Office Manager
- Opens PDFs from their computer in the web UI
- Annotates documents with stamps, text, arrows, shapes, signatures
- Burns annotations into the PDF and saves the result back to their computer
- Annotations auto-save in the browser (never loses work during a session)

### Form Designer
- Creates reusable form templates with field positions, types, and validation rules
- Drag-and-drop field positioning on the PDF canvas
- Reuses templates across many submissions

### Form Filler / Field Worker
- Opens a form template and fills in data
- Saves drafts automatically (never loses work)
- Submits completed forms for processing
- Works from phone/tablet in the field (v2 — mobile optimization)

### Vertical App Developer
- Integrates PDF processing via REST API
- Creates templates programmatically
- Sends PDF bytes + submission data, receives generated PDF bytes back
- Listens for events (pdf.form.submitted, pdf.document.generated)

## Design Principles

1. **No file storage.** The editor is a processing engine. It receives PDF bytes, processes them, and returns the result. The caller decides where files live — local disk, S3, network share, whatever. The editor doesn't care and doesn't store.

2. **Standalone before integration.** The module must work with its own web UI before any other app exists. A user can open a PDF from their computer, annotate it, and save it back — no server-side storage needed.

3. **No lost work.** Annotations auto-save in the browser (Zustand store). Form submission drafts persist in Postgres across sessions. Work is never silently lost.

4. **Templates and submissions in Postgres, files nowhere.** Form templates (field layouts, validation rules, positions) and form submissions (field data, draft/submitted status) persist in the database. PDF files are never stored by the editor.

5. **Simple status machine.** draft → submitted for v1. Approval workflows, multi-person signing chains, and interactive PDF form fields are v2.

6. **Platform-compliant.** Multi-tenant via tenant_id, Guard→Mutation→Outbox pattern, EventEnvelope on NATS, standard health/metrics endpoints. Plugs into the platform like every other module.

## MVP Scope

### In MVP (v0.1.x)

- Render annotations onto a PDF: receive PDF bytes + annotation JSON, return annotated PDF bytes
- Form templates: define fields on a PDF, reuse across submissions
- Form filling: submit data, validate, store as draft, submit when ready
- PDF generation: receive PDF bytes + submission ID, return filled PDF bytes
- Auto-save drafts (form submissions in Postgres, annotations in browser)
- Standalone web UI: open PDF from computer, annotate, save back to computer
- CORS configuration for standalone frontend
- PDF validation on processing requests (magic bytes check, size limit)
- Events: pdf.form.submitted, pdf.document.generated
- OpenAPI contract for API consumers

### NOT in MVP

- Server-side PDF file storage (explicitly out of scope — caller manages files)
- Mobile-optimized experience (responsive layout, touch-friendly annotation)
- Multi-person signing/approval chains
- Interactive PDF form fields (AcroForm / XFA)
- Password protection / PDF encryption
- Integration with other platform modules via events (API works, event consumers come later)
- CRDT-based concurrent editing

## Data Model

### Tables

**form_templates**
- id, tenant_id, name, description, created_by, created_at, updated_at

**form_fields**
- id, template_id (FK), field_key, field_label, field_type (text|number|date|dropdown|checkbox), validation_rules (JSONB), pdf_position (JSONB — x, y, width, height, page), display_order

**form_submissions**
- id, tenant_id, template_id (FK), submitted_by, status (draft|submitted), field_data (JSONB), created_at, submitted_at

**events_outbox**
- Standard platform outbox table for EventEnvelope publishing

### What is NOT stored

- **PDF files** — never stored by the editor. Caller provides bytes, editor processes, caller saves result.
- **Annotations** — live in the browser (Zustand annotationStore). Server only receives them for rendering into a PDF.
- **Generated PDFs** — returned as response bytes. Caller decides where to store them.

## Structural Decisions

### SD-1: One module, not two
**Decision:** Annotations and forms live in the same module.
**Rationale:** They share the same rendering pipeline (pdfium-render). Splitting would create cross-module dependencies. Single deployment, single database, shared code.
**Source:** Grok stress test (2026-02-24)

### SD-2: No file storage — stateless processing engine
**Decision:** The editor never stores PDF files. It receives PDF bytes, processes them (burn annotations, fill form data), and returns the result. The caller manages file storage.
**Rationale:** User requirement. "The PDF editor should not be concerned with where [the file] originates." In standalone mode, the user opens from their computer and saves back. In API mode, the calling app provides bytes and stores the result wherever it wants. This eliminates S3/MinIO infrastructure dependency and keeps the module focused on processing.
**Source:** User requirement (2026-02-24). Replaces original SD-2 (S3 object storage).

### SD-3: pdfium-render for PDF manipulation
**Decision:** Use pdfium-render Rust crate for reading, overlaying, and writing PDFs.
**Rationale:** Active crate (v0.8.37, Feb 2026), supports high-level edit/annotate/form-fill on existing PDFs. The Node.js version used pdf-lib and pdfkit — pdfium-render is the closest Rust equivalent.
**Gotchas:** PDFium binary must be bundled (dynamic load), thread_safe feature required for concurrent use, watch for lifetime quirks on long-lived document handles. Keep handles short-lived per request.
**Source:** Grok stress test (2026-02-24)

### SD-4: Simplified status machine for v1
**Decision:** Form submissions use draft → submitted only. No approved/rejected states.
**Rationale:** Approval workflows require multi-person chains, permissions, and notification integration. That's v2 complexity. For v1, the user submits and the consuming app decides what to do with it.
**Source:** Grok stress test (2026-02-24)

### SD-5: Annotations live in the browser
**Decision:** Annotation data lives in the browser's Zustand annotationStore. There is no server-side annotation persistence.
**Rationale:** With no file storage, there's no document to associate annotations with on the server. The user annotates in the browser and either burns annotations into the PDF (server-side render) or saves the annotated PDF to their computer. Autosave is browser-local.
**Source:** Architecture overhaul (2026-02-24). Replaces original SD-5 (optimistic locking on annotations).

### SD-6: PDF validation on processing requests
**Decision:** Processing endpoints validate incoming PDF bytes: magic bytes (%PDF-), configurable size limit (50MB default).
**Rationale:** Prevents non-PDF payloads, PDF bomb attacks, and memory exhaustion.
**Source:** Grok adversarial review (2026-02-24). Updated from "upload validation" to "processing request validation".

### SD-7: Standalone + integration dual mode
**Decision:** Module runs with its own React web UI for standalone use AND exposes REST API for app integration.
**Rationale:** Owner needs to use the PDF editor today (approving POs) before Fireproof ERP is complete. Other apps need API access. CORS is configured for standalone frontend.
**Source:** User requirement (2026-02-24)

### SD-8: Frontend kept as-is, API repointed
**Decision:** Existing React + Vite + TypeScript + Zustand frontend is preserved. Only API calls are changed to point at the new Rust backend.
**Rationale:** Frontend works today for annotations. Rewriting it would double the effort without adding value. Create a typed API client layer, then update stores.
**Source:** Planning decision (2026-02-24)

## Open Questions

1. **pdfium-render annotation type coverage:** Which annotation types (stamps, arrows, shapes, signatures, highlights) can pdfium-render overlay reliably? Need to test each type early in bd-1647. Fallback to lopdf for unsupported types.

2. **Font embedding for form generation:** When rendering text onto PDFs from form data, which fonts are available? Need to bundle a base font set or rely on PDF's embedded fonts.

3. **Frontend test strategy:** The existing frontend has minimal tests. Should bd-37tz (smoke tests) use Playwright, or is a simpler approach sufficient?

4. **Signature handling:** Signatures in the annotation system are likely Base64 PNG images. How do we overlay PNG onto PDF via pdfium-render? This is the riskiest annotation type.

5. **Max PDF size for in-memory processing:** Without file storage, PDFs are processed entirely in memory (receive bytes → process → return bytes). What's the practical upper limit before the server runs out of memory? The 50MB size limit needs validation against actual memory usage with pdfium-render.
